# Pre-Design：决策卡片生成器

`pre_design` 是 OpenSpec 工作流的"想清楚"阶段，产出**决策卡片**——给用户对齐，给下游 `ff` / `ff-big` 做约束。

它做四件事：

1. **理解需求** — 弄清"为什么做、做什么、不做什么、什么算成功"
2. **选路线** — 在 2-3 条思路级方向里拍板（用 SDK 还是自研、同步还是异步等）
3. **立护栏** — 输出下游必须遵守的硬约束 + 禁止脑补的边界
4. **指方向** — 根据复杂度推荐下一步走 `/opsx:ff` 或 `/opsx:ff-big`

**它不做的事：**

- 不展开架构（模块切分、数据流、接口签名 → 都是 design.md 的职责）
- 不指挥下游每个 artifact 写什么（OpenSpec schema 已定义）
- 不在文档里堆元规则（"中约束精确定义"等给 AI 看的护栏不写进产物）

**Input**: `/opsx:pre-design` 后面可以跟需求主题、change 名称，或一段问题描述。

<HARD-GATE>
在 `pre_design.md` 写完并经用户确认之前：
- 不要生成 `proposal.md`、`design.md`、`tasks.md`、`specs/`
- 不要调用 `/opsx:ff`、`/opsx:ff-big`、`/opsx:new`、`/opsx:continue`、`/opsx:apply`
- 不要编写任何实现代码

写完后直接结束，告知用户文件位置，由用户决定下一步。
</HARD-GATE>

## 核心原则

- **对话可充分展开，文档只记结果** — 方案对比在对话里讨论透，pre_design.md 里只写"选了什么 + 一两句放弃理由"
- **每次一个问题** — 不要一条消息抛多个问题；优先选择题
- **路线选择 vs 方案展开** — 选哪条路在 pre_design；这条路怎么走在 design.md
- **候选默认结构化展开** — 给候选时默认按"主要权衡（按维度）+ 在本项目的具体后果"展开，让用户看清取舍后再选；技术细节、伪代码、ASCII 图按需追加，不要默认堆
- **用户表达观点时进入讨论模式** — 用户不只是选 A/B/C，而是给出自己的判断、补充、反对或新方案时，把它当**讨论**而不是**指令**：先客观评价（哪里对、哪里有保留），再决定是否调整推荐；坚持原推荐时给出依据，不为顺从而改口
- **没有明显最优时不强凑** — 列出决策关键点让用户拍板，不要为了"显得专业"硬推某个方向

## 流程（4 步）

1. **探索项目上下文** — 检查相关文件、活跃 change、近期 commit；如有 `/opsx:explore-req` 交接摘要则直接消费
2. **澄清需求与边界** — 每次一个问题，聚焦目的 / 约束 / 成功标准 / 非目标
3. **提出 2-3 条路线 + 推荐** — 思路级判断，不是模块级展开；先亮推荐与理由，再说放弃了什么
4. **创建/复用 change，写入 `pre_design.md`，告知下一步** — 默认单文件；建议走 `/opsx:ff` 或 `/opsx:ff-big`

终止状态：`pre_design.md` 已写入并告知用户。**不自动调用任何后续 command。**

## 流程详述

### 1. 探索项目上下文

**必须读取 OpenSpec 配置**：在写入任何 OpenSpec 产物前，读取 `openspec/config.yaml`，提取适用于 pre_design 和所有 OpenSpec 产物的语言、格式与内容规则；若配置要求中文，`pre_design.md` 的解释性内容必须使用中文，必要技术标识、文件路径、命令、代码符号和结构性英文标题可保留原文。

**有交接摘要**：跳过摘要已确认的部分（需求定位、实现现状、依赖就绪度、范围判断），只补充未覆盖的内容。摘要中"未决问题"和标注"待 pre-design 澄清"的字段优先处理。

**无交接摘要**：做轻量探索，目标是判断：当前需求是否适合单个 change、仓库里是否有相关活跃 change、现有结构对本次设计有什么影响。

### 2. 澄清需求与边界

每次只问一个问题，优先用选择题，聚焦：

- 目的 / 触发原因
- 硬约束 / 不变量
- 成功标准
- 非目标
- 路线偏好（如果用户已有倾向）

**进入步骤 3 之前的硬门（必须满足）**：用户已**显式确认**以下三项：

- **目的 / 触发原因**
- **Non-Goals**（明确不做什么）
- **硬约束 或 成功标准**（至少其一）

如果初始输入里已经隐含某项，**复述并请用户确认**（"我理解你的目的是 X，对吗？"），不要默认通过。任一未明 → 继续问，**不进步骤 3**。

> "路线偏好"不进硬门——用户没倾向也是合理状态。

**范围失控的处理**：如果发现需求实际覆盖多个独立问题，先帮助用户拆分（哪些应成独立 change、关系是什么、当前先做哪个），不要继续细化一个过大的 change。

### 3. 提出 2-3 条路线 + 推荐

边界基本清楚后，提出 2-3 条**路线级方向**——是"用 SDK 还是自研"、"同步还是事件驱动"、"单体还是拆服务"这种思路级判断，**不是模块切分、接口签名、数据流图**。

**每条路线展开为结构化小节（约 10-15 行）**，固定包含：

- **一句话定位** — 用户能凭一句话区分
- **要点** — 这条路具体怎么走（1-2 行；保持路线级别，不展开模块/接口）
- **主要权衡（按维度，没代价的维度直接省略）**：
  - 性能 / 复杂度：具体代价
  - 维护成本：具体代价
  - 扩展性：具体代价
  - 风险：具体代价
- **在本项目的具体后果**：
  - 短期（本次实现）：会发生什么
  - 中期（下一两个 change）：会撞上什么
  - 已知风险点：结合现有代码 / 约束 / 团队习惯

> 技术细节展开（具体技术选型方案）、伪代码、ASCII 图、类比等**按需追加**——用户点名要才给，避免越界到 design.md 的职责。
>
> **节奏**：先亮推荐路线 + 一句话理由，再依次展开候选；不要把所有候选一次甩成几屏长文。

**推荐策略（区分两种情况）**：

- **有明显最优时**：先亮推荐路线 + 一句话理由（结合**当前项目的具体约束**，不是抽象偏好），再说放弃了什么 + 一两句为什么
- **势均力敌时**：显式说**"这两条都能走，差别在 X / Y，需要你拍板"**，列出决策关键点交给用户；不要为了"显得专业"硬推某个方向

**用户表达观点时（不只是选 A/B/C）**：

把用户的判断 / 补充 / 反对 / 新方案当**讨论**，按以下结构回应：

1. **哪里对** — 这条想法在哪些维度站得住、和现有约束如何契合（具体，不空泛）
2. **哪里有保留** — 风险点、边界条件、被忽略的代价
3. **要不要调整推荐** — 采纳 / 部分采纳 / 仍坚持原推荐；坚持时给出依据，不为顺从而改口

**反模式**：

- 三条路线只是复杂度递增的梯度（简单版 / 中等版 / 完整版）——应该是思路不同
- 为了凑数硬编第三条——只有两条有意义就只提两条
- 在第 2 步问清楚之前就甩详细方案——背景没对齐，方案就是空中楼阁
- 确实有明显最优却回避结论——用户需要的是判断
- 在推荐里回避取舍——必须坦诚说出代价
- 用户提观点 → 直接"好按你说的"，没评估
- 用户提观点 → 只挑毛病，没承认对的部分
- 用户坚持 → 立场漂移翻盘——推荐变化必须有新信息支撑

### 4. 创建/复用 change，写入文档，告知下一步

**创建或复用 change**：

- 路径：`openspec/changes/<type>-<topic>/`
- `type ∈ {feature, fix, refactor, research, docs}`
- `topic` 用 kebab-case，只表达主题，不塞实现细节
- 通过 OpenSpec CLI 创建，确保 `.openspec.yaml` 存在
- 已有**明确匹配**的非归档 change 可复用（主题、目标、范围都对齐，不是表面相似）；多候选时停下问用户

**落盘前向用户预览整体方案并等待确认（硬门）**：

预览前必须自检：整体方案和将要写入的 `pre_design.md` 必须符合 `openspec/config.yaml`；若配置写明“所有 OpenSpec 产物必须使用中文”，则解释性内容必须使用中文，必要技术标识、文件路径、命令、代码符号和结构性英文标题可保留原文。

向用户输出形如：

```
落盘前请确认整体方案：

【推荐路线】<一句话定位>
【关键决策】
  - <决策 1>：<理由>
  - <决策 2>：<理由>
  - ...
【护栏】
  Must follow：<3-5 条>
  Forbidden：<3-5 条>
【下一步】<推荐 /opsx:ff 还是 /opsx:ff-big + 一句理由>

OK 落盘？还是要调整哪一块？
```

- 任何"调整"/"不对"/"再想想" → 回到对话，**不写文件**
- 明确"OK / 落盘吧 / 写吧"才进入"写入 pre_design.md"
- 用户改主意时不要硬把已有讨论塞进去；按修订意见回到步骤 2 或 3 继续

**写入 pre_design.md**：

- 默认单文件 `pre_design.md`
- 不要分卷；如果内容明显放不下，说明范围太大，建议拆 change 而不是分卷

**告知下一步**：

- 在 `## Next step` 节写明推荐走 `/opsx:ff` 还是 `/opsx:ff-big`
- 一句话说明判断理由（复杂度信号）

## pre_design.md 内容（4 节 + 1 附属）

```markdown
# Pre-Design: <change-name>

## 1. Problem & Goals
- 真实要解决的问题（含背景与触发原因）
- Goals（本次要达成什么）
- Non-goals（明确不做什么 / 看起来相关但本次不做）

## 2. Constraints / Invariants
- 硬约束（必须遵守的边界）
- 不变量（不应被下游推翻的事实）

## 3. Direction & Key decisions
- 推荐路线：<一句话定位>
- 放弃路线：<一句话> —— 理由：<一两句>
- 关键决策（每条一句话 + 理由）：
  - <决策 1>：<理由>
  - <决策 2>：<理由>
  - ...

## 4. Guardrails for downstream
- Must follow（3-5 条，下游必须遵守的硬条目）
- Forbidden to invent（3-5 条，下游禁止脑补的内容）

## Next step
- 建议下一步：`/opsx:ff` 或 `/opsx:ff-big`
- 复杂度信号：<一句话理由>
```

**写作约束**：

- 每节内容**只记结果**，不复述对话过程
- 关键决策**每条一句话**，理由**最多两句**——超出说明决策没真正拍板，回到对话澄清
- 不写 Architecture direction、Critical flows、模块切分、接口签名、数据流图——这些是 design.md 的职责
- 不写 OpenSpec mapping（"proposal 该写什么"等）——schema 已定义

## 消费交接摘要

如果对话中存在 `/opsx:explore-req` 的交接摘要（以 `## 交接摘要：` 开头）：

| 摘要字段 | 处理 |
|---------|------|
| 需求定位 / 实现现状 / 依赖就绪度 / 范围判断 | 直接继承，跳过对应探索 |
| 已确认的目标、成功标准、范围内/外 | 作为已知基线，澄清时只问未覆盖维度 |
| 推荐实现路径 | 作为首选路线起点 |
| 未决问题 / "待 pre-design 澄清" | 优先澄清队列 |

无摘要时正常走完整流程。

## 失败保护

出现以下情况就暂停澄清，不要硬写文档：

- 范围明显过大，无法用单个 change 承载
- 多个活跃 change 候选无法唯一判断
- 关键路线决策尚未形成
- `type` 或 `topic` 无法稳定命名

宁可多问几轮，也不要产出方向含糊的 `pre_design`。

## 完成条件

- 已完成需求梳理与路线选择
- 已确定并创建/复用目标 change，根目录存在 `.openspec.yaml`
- 已写入 `pre_design.md`（4 节 + Next step）
- 已告知用户文件位置和推荐的下一步 command
- 在此结束，不自动进入后续流程

## 明确不做的事

- 不 commit 文档
- 不在 `archive/` 下创建或修改 change
