# A 股深度投研 Obsidian 工作流设计

## 背景与目标

本项目用于每日 A 股股票池深度分析。用户通常只提供股票名称和代码，例如 `炼石航空(000697.XSHE)`。Claude Code 需要基于当前数据、联网检索和本地辅助脚本，为每只股票生成独立长篇深度分析文档，并在所有个股分析完成后生成每日综合文档。

第一版目标不是构建全自动投研系统，而是建立一个可重复、可审计、可逐步增强的 Claude Code 主导工作流：流程固定、输出稳定、来源可追溯、适合 Obsidian 长期管理，并严格避免把模型训练数据当作当前事实依据。

## 总体架构

第一版采用：

```text
CLAUDE.md
  项目级硬规则与边界

.claude/skills/stock-analysis/
  固定股票分析工作流

scripts/
  本地辅助脚本：解析输入、创建目录、读取配置、拉取 Tushare/akshare 数据、生成模板

templates/
  Obsidian Markdown 模板

companies/
  公司长期档案

stock/YYYY-MM-DD/
  每日股票池分析产物
```

各部分职责：

- `CLAUDE.md`：项目宪法，写任何时候都必须遵守的硬规则和触发 skill 的要求。
- `stock-analysis` skill：写股票分析任务的完整执行流程，避免每次临场发挥。
- `scripts/`：负责机械化动作，例如解析股票列表、创建目录、生成模板、获取结构化数据底稿。
- `templates/`：保证 Markdown 文件结构、frontmatter、标题层级、来源清单和 Obsidian 链接一致。
- Claude Code：负责联网检索、来源分级、事实核验、深度分析、风险识别、综合排序和最终写作。

第一版不做 MCP，也不做自定义多 agent 系统。后续当流程稳定、股票数量增加或耗时明显上升时，再考虑事实核查 agent、个股分析 agent 或行业综述 agent。

## 输出目录与文件结构

默认使用当前日期创建每日目录，除非用户明确指定日期。

```text
stock/YYYY-MM-DD/
  index.md
  000697.XSHE-炼石航空-日评.md
  000791.XSHE-甘肃能源-日评.md
  002428.XSHE-云南锗业-日评.md
  当日股票池排序.md
  当日主题与行业综述.md
  当日研究日志.md
```

公司长期档案放在：

```text
companies/
  000697.XSHE-炼石航空.md
  000791.XSHE-甘肃能源.md
```

首次遇到某只股票时，同时创建公司长期档案和当天日评。若公司档案已存在，则读取并参考，不重复创建。

## 每日入口页

`stock/YYYY-MM-DD/index.md` 是当天 Obsidian MOC，包含：

- 当天股票池列表
- 每只股票的日评链接
- 每只股票的关注等级
- 三份综合文档链接
- 共同主题/行业标签
- 今日核心结论
- 后续待跟踪事项

## 个股日评结构

个股日评采用长篇深度版，但结论前置。报告重点放在前面，不重要的细节和资料放在后面。

建议结构：

1. 投研结论摘要
2. 关注等级
3. 核心逻辑
4. 最大风险
5. 需要继续跟踪的变量
6. 公司与业务概况
7. 行业与政策背景
8. 财务与经营质量
9. 估值与同业比较
10. 技术面与资金情绪
11. 最新公告、新闻与事件催化
12. 信息来源与可信度分级
13. 无法确认或证据不足事项
14. 建议写入公司档案的信息

关注等级采用 5 档：

1. 重点关注
2. 积极关注
3. 谨慎关注
4. 观察跟踪
5. 暂不关注

报告只输出研究观点，不提供具体买入价格、卖出价格、止盈止损或仓位建议。

## 公司长期档案维护

公司长期档案只沉淀稳定、长期有效的信息，包括：

- 主营业务
- 股权结构和治理
- 核心资产、项目、产能
- 产业链位置
- 长期财务特征
- 长期风险
- 历史重大事项
- 相关日评反向链接

长期档案维护采用保守维护加人工确认建议的混合方式：

- 明显稳定、长期有效、来源可靠的信息可以写入公司档案。
- 中短期、未充分确认或容易过时的信息不直接写入长期档案。
- 每篇日评末尾保留 `建议写入公司档案的信息`，供用户后续确认。

## 每日综合文档

每日生成三份综合文档：

1. `当日股票池排序.md`
   - 汇总关注等级、排序、核心理由、风险差异。

2. `当日主题与行业综述.md`
   - 分析股票池背后的共同主题、行业链条、政策催化和市场叙事。

3. `当日研究日志.md`
   - 记录检索来源、数据限制、无法确认事项和后续跟踪问题。

## 数据来源与事实校验

每次分析必须基于当前数据和联网检索，不得用模型训练数据作为当前事实依据。

来源分级：

- A 级：公司公告、交易所、巨潮资讯、监管/政府网站、定期报告。
- B 级：公司官网、投资者关系、主流财经媒体、可靠数据平台、Tushare、akshare。
- C 级：财经门户二次报道、互动平台、社区讨论、市场传闻。
- 未确认：只有单一非权威来源，或搜索结果无法交叉验证。

使用规则：

- A/B 级来源可作为事实底座。
- C 级来源只能用于市场情绪、题材线索或待验证信息，不能单独支撑核心结论。
- 未确认信息必须明确标注“未确认”或“不足以支撑判断”。
- 如果信息冲突，优先采用 A 级来源，并在报告中说明冲突。
- 公司长期档案只吸收 A/B 级且长期有效的信息。

正文采用段落内少量来源加文末来源清单的方式。关键事实在正文标注来源或数据日期，文末统一列出来源、检索时间和无法确认项。

## 默认分析流程

当用户输入股票池或要求分析 A 股股票时，应触发 `stock-analysis` skill，并按顺序执行：

1. 解析输入股票列表。
2. 确定日期和输出目录。
3. 调用初始化脚本生成模板。
4. 调用数据脚本获取 Tushare/akshare 结构化底稿。
5. 读取已有公司长期档案。
6. 联网搜索最新公告、新闻、政策、行业和行情背景。
7. 按 A/B/C/未确认做来源分级。
8. 逐只生成长篇日评。
9. 保守更新公司长期档案。
10. 生成三份综合文档。
11. 生成每日入口页。
12. 做一致性和来源检查。

## 本地配置

Tushare token 使用本地配置文件，不提交到 git。例如：

```text
.env
```

或：

```text
.claude/settings.local.json
```

规则：

- token 不写入报告。
- token 不写入 git。
- `.gitignore` 必须排除 `.env` 等敏感文件。
- 脚本优先读取本地配置，必要时再读取环境变量。

## 本地脚本

第一版建议实现：

```text
scripts/
  parse_stock_list.py
  init_daily_reports.py
  fetch_stock_data.py
```

### parse_stock_list.py

将用户粘贴的股票列表解析成结构化列表。

输入示例：

```text
1. 炼石航空(000697.XSHE)
2. 甘肃能源(000791.XSHE)
```

输出示例：

```json
[
  {"name": "炼石航空", "code": "000697.XSHE"},
  {"name": "甘肃能源", "code": "000791.XSHE"}
]
```

### init_daily_reports.py

负责：

- 创建 `companies/`
- 创建 `stock/YYYY-MM-DD/`
- 检查公司长期档案是否存在
- 首次出现时创建公司档案模板
- 创建每日个股日评模板
- 创建 `index.md`
- 创建三份综合文档模板

### fetch_stock_data.py

负责调用 Tushare/akshare 拉基础数据底稿。

数据源策略：

- Tushare 作为优先结构化数据源。
- akshare 作为补充和 fallback。
- 联网搜索仍是最新公告、新闻、政策、行业和市场叙事的必要步骤。

可拉取字段包括：

- 股票基础信息
- 近期行情
- 财务摘要
- 估值指标
- 成交量、换手率等交易数据
- 行业分类、概念、资金流等接口可用字段

脚本输出保存为：

```text
stock/YYYY-MM-DD/data/
  000697.XSHE.json
  000791.XSHE.json
```

每个 JSON 应记录字段来源、数据日期和缺失项。拉不到的数据写入 `missing_fields`，不阻塞报告生成。

## CLAUDE.md 内容边界

`CLAUDE.md` 只放项目硬规则和 skill 触发要求，例如：

- 项目是 A 股深度投研与 Obsidian 知识库。
- 用户通常只提供股票名称和 A 股代码。
- 当用户输入股票池或要求分析 A 股股票时，必须调用 `stock-analysis` skill。
- 每次分析必须联网搜索最新信息。
- 不得用模型训练数据作为当前事实依据。
- 必须优先使用 A/B 级来源。
- Tushare/akshare 数据只能作为结构化底稿，不能替代最新搜索。
- 所有当前事实必须标注数据日期或检索时间。
- 报告只构成研究观点，不给买卖价位、仓位、止盈止损。
- 输出必须写入 `companies/` 和 `stock/YYYY-MM-DD/`。
- 报告必须 Obsidian 友好：frontmatter、标签、双链、反向链接。
- token 和本地配置不得提交。

## stock-analysis skill 内容边界

`stock-analysis` skill 放完整流程、检查清单、输出要求和质量门槛。

触发场景：

- 用户输入 A 股股票池。
- 用户要求分析某只或多只股票。
- 用户要求生成每日股票分析文档。

skill 必须规定执行顺序、来源分级、报告结构、最终检查项，并要求 Claude Code 在完成前核对所有文件和来源。

## 质量控制

每次分析完成前必须检查：

### 文件完整性

- 每只股票都有日评。
- 首次出现的股票有公司长期档案。
- 每日目录有 `index.md`。
- 每日目录有三份综合文档。

### 来源完整性

- 每只股票日评都有来源清单。
- 关键事实有来源或数据日期。
- 无法确认事项被单独列出。
- C 级来源没有单独支撑核心结论。

### 结论边界

- 有 5 档关注等级之一。
- 没有具体买卖价格。
- 没有仓位建议。
- 没有止盈止损建议。

### Obsidian 兼容

- 有 YAML frontmatter。
- 有 tags。
- 有公司档案双链。
- 每日入口页链接到所有产物。

### 长期档案维护

- 只写入稳定、长期有效的信息。
- 中短期信息放入日评的 `建议写入公司档案的信息`。
- 公司档案链接相关日评。

## 技术验证

第一版实现完成后，应验证：

- 股票列表解析是否正确。
- 日期目录是否正确创建。
- 模板文件是否正确生成。
- `.env` 或本地配置不会被提交。
- Tushare token 能被读取。
- Tushare 不可用时 akshare fallback 是否能工作。
- 数据 JSON 是否包含来源和数据日期。
- 生成的 Markdown 在 Obsidian 中结构清晰。

## 后续扩展

第一版稳定后，可以逐步增加：

- 更丰富的 akshare/Tushare 字段。
- 公告抓取脚本。
- 来源去重与可信度自动标注。
- 公司长期档案增量更新工具。
- 多股票并行分析 agent。
- 独立事实核查 agent。
- 定期复盘报告。
- 行业/主题长期档案。
