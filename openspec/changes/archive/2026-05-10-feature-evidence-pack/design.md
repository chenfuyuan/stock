## Context

当前股票分析流程需要把结构化抓取结果提供给 Claude Code 作为分析材料。全量历史行情、资金流等高频数据在原始 JSON 中可能快速膨胀，直接作为默认输入会占用大量上下文，并且容易掩盖真正需要审阅的关键事实。

本设计采用 raw-first 路线：原始 JSON 先作为本地事实源落盘，再从原始 JSON 派生面向分析任务的小型 Evidence Pack。Evidence Pack 是 Claude Code 的默认读取材料，原始 JSON 仅用于审计、复算和必要的深入检查。

## Goals / Non-Goals

**Goals:**

- 保留原始 JSON 作为可复算、可审计的本地事实源。
- 让 Claude Code 默认读取 Evidence Pack，而不是完整原始 JSON。
- 高频行情和资金流默认限制为最近一年窗口。
- 财务原始数据允许全量保留，但 Evidence Pack 中只能按任务选择或聚合。
- 当前事实在 raw 与 Evidence Pack 链路中保留数据日期、抓取时间、来源、状态和 `raw_path` 等元数据。
- 提供小型 JSON fixture 或 shape 示例，用于验证字段结构和派生逻辑。

**Non-Goals:**

- 不构建完整因子平台。
- 不引入数据库化投研系统。
- 不重写现有报告模板。
- 不把完整原始 JSON 作为 Claude Code 默认 prompt 或输入材料。
- 不用模型训练数据替代当前抓取数据。

## Decisions

### 1. 使用 raw-first 持久化，而不是抓取时只写固定摘要

原始抓取结果先保存为 raw JSON，Evidence Pack 从 raw JSON 派生生成。这样可以在不重新联网抓取的情况下复算 Evidence Pack，也能在分析结论需要追溯时回到完整事实源。

替代方案是抓取时只写固定摘要 JSON。该方案上下文更小，但会把当前分析视角固化到数据层，后续复算、审计或深入检查缺少原始依据，因此不采用。

### 2. 将 Evidence Pack 设为默认分析输入，而不是依赖人工选择

分析命令、工作流说明和测试应围绕 Evidence Pack 建立默认路径。原始 JSON 可以存在本地，但默认不进入 Claude Code 输入材料，避免上下文控制依赖每次人工提醒。

替代方案是每次分析时临时选择读取 raw 或编写解析代码。该方案灵活，但容易遗漏窗口控制，也难以稳定保证上下文大小，因此不采用。

### 3. 高频数据默认使用一年窗口，财务数据按任务选择或聚合

行情、资金流等高频 payload 默认只保留或进入 Evidence Pack 最近一年数据。财务 raw JSON 可以保留完整历史序列，但 Evidence Pack 不整块复制全量财务原始数据，只保留任务相关的关键年度、季度、指标摘要或聚合结果。

替代方案是对所有数据统一截断。该方案简单，但会破坏财务长期周期、拐点和历史对比能力，因此不采用。

### 4. Evidence Pack 必须携带可追溯元数据

每个 Evidence Pack 至少记录 `raw_path`、生成时间、数据日期或抓取时间、来源和状态信息。这样 Claude Code 使用小型输入时仍能确认事实新鲜度、来源范围和派生依据。

替代方案是只保留业务字段。该方案更短，但会削弱当前事实要求和审计能力，因此不采用。

### 5. 用小型 fixture 或 shape 示例覆盖结构，不携带完整历史

测试和开发示例使用小型 JSON fixture 或 shape 示例来表达字段结构、元数据和典型数据窗口。fixture 不承担完整历史事实保存职责，避免测试材料本身膨胀。

替代方案是复制真实完整 raw JSON 作为 fixture。该方案真实度高，但会制造大文件和敏感配置误入风险，因此不采用。

## Risks / Trade-offs

- Evidence Pack 过度压缩导致分析遗漏 → 保留 `raw_path` 并确保可从 raw JSON 无需联网复算，必要时允许回查本地 raw。
- 一年窗口不足以覆盖特殊长期行情问题 → 默认保持一年窗口，只有在后续需求明确扩大窗口时再调整。
- 财务数据选择或聚合规则不清导致结果不稳定 → 在 spec 中约束 Evidence Pack 不得整块复制全量财务 raw，并在任务中落实最小可测派生逻辑。
- raw JSON 本地落盘可能误被放入 prompt 或提交 → 工作流和测试明确默认输入为 Evidence Pack，并在收尾自检中确认 token、`.env` 和本地私密配置未被保存到产物。
