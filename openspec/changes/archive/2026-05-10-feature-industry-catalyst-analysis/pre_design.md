# Pre-Design: feature-industry-catalyst-analysis

## 1. Problem & Goals
- 真实要解决的问题：现有个股分析容易停留在单只股票视角，行业背景、同催化股票对比和跨日期复用沉淀不足，导致后续分析难以复利使用。
- 触发原因：用户希望个股分析结合行业进行判断，对比同行业或同一催化相关股票，并将行业层面的分析也沉淀到 Obsidian 知识库中。
- Goals：
  - 建立行业/同催化横向比较的报告框架，使个股分析能呈现共同驱动、相对差异和共同风险。
  - 将行业/催化分析作为长期知识库节点沉淀，支持被后续个股报告引用、增量更新和复用。
  - 保持现有 `vault/companies/` 与 `vault/stock/<date>/` 工作流可用，不要求迁移到全新入口。
- Non-goals：
  - 不做自动行业分类、概念归属或同催化股票池生成。
  - 不做量化评分、因子模型、排名模型或买卖价位建议。
  - 不用行业分析替代个股深度分析；行业/催化节点只提供上下文、比较框架和复用沉淀。

## 2. Constraints / Invariants
- 所有当前事实必须使用当前数据和 web search，并包含来源、发布日期或检索时间、数据日期。
- 来源优先级延续项目规则：优先使用公告、交易所、CNINFO、监管/政府、公司 IR、主流财经媒体、可靠数据平台、Tushare 和 akshare。
- 输出必须保持 Obsidian 友好：YAML frontmatter、tags、wikilinks、backlinks、清晰标题和结论先行。
- 行业/催化长期节点必须能被个股日评双向链接，并支持后续增量更新。
- 不改变投资研究免责声明边界：不得提供具体买入价、卖出价、止盈止损或仓位建议。

## 3. Direction & Key decisions
- 推荐路线：行业/催化长期节点 + 日评引用；把行业或同催化研究做成稳定 Obsidian 知识节点，个股日评记录本次增量、横向比较结论和到长期节点的 wikilink。
- 放弃路线：仅在日评内嵌扩展行业比较 —— 理由：实现快但行业判断仍散落在每日文档里，无法满足可复用沉淀的核心成功标准。
- 放弃路线：结构化行业数据层优先 —— 理由：扩展性强但会引入分类口径、数据源冲突和模型化设计，超出本次“只做框架”的边界。
- 关键决策：
  - 行业/催化分析作为长期知识库一等节点：因为本次成功标准优先是可复用沉淀，而不是单次报告更丰富。
  - 个股日评引用长期节点并写增量：避免行业结论散落，同时保留当日判断上下文。
  - 同行业与同催化共用一套横向比较框架：因为实际研究对象是共同驱动因素下的股票，不应只限于传统行业分类。
  - 本次只定义框架、模板、索引和写入规则：避免把自动分类、评分模型或投资建议混入范围。
  - 横向比较必须带证据链：因为行业判断和催化归因如果不可追溯，后续复用会放大错误。

## 4. Guardrails for downstream
- Must follow：
  - 行业/催化节点必须能被后续个股报告双向链接和增量更新。
  - 横向比较必须覆盖对比对象、共同催化/行业逻辑、差异点、共同风险和证据来源。
  - 行业/催化结论不得替代个股深度分析，只能作为上下文、比较和复用沉淀。
  - 保持现有 `vault/companies/`、`vault/stock/<date>/` 和长期知识库入口兼容。
  - 所有当前事实必须使用当前数据和 web search，不得基于模型训练数据。
- Forbidden to invent：
  - 不脑补自动行业分类、概念归属或同催化股票池。
  - 不设计量化/因子评分、排名模型或买卖价位建议。
  - 不要求迁移现有 `vault/companies/` 与 `vault/stock/<date>/` 工作流。
  - 不把行业分析只写进日评而没有长期沉淀入口。
  - 不在本 change 中展开或锁定模块切分、接口签名、数据流图等 design.md 职责内容。
